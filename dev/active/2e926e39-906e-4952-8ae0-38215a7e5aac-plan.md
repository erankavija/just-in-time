## 2. Technical soundness and architectural fit

- **Approach:** The work is a set of independently landable conformance and readability fixes over the existing CLI, gate, validation, and skill surfaces. Almost every change extends an existing primitive rather than introducing a new subsystem. The one cross-cutting backbone is REQ-06: a written grammar standard plus a tree-wide rename sweep that the specific first-guess fixes conform to. CLI changes favor clean canonical renames with no compatibility shims (D2); the consumer set is entirely ours (MCP regenerates from the CLI schema, ../gf2 and the jit-* skills are local), so renames are absorbed by updating consumers in the same change (D4).

- **Reuses / integrates with:**
  - Query default routing reuses the shared `run_query_all` helper (main.rs:689, filters applied main.rs:705-710, dispatched main.rs:3461-3470, reused main.rs:1863): a bare `jit query` calls it with no filters; `ready` is a visible alias of the existing `Available` variant (cli.rs:1483-1499). No new query semantic and no traversal; `query all` keeps its existing filters and has no `--type` filter.
  - Type-flag validation reuses the config `[type_hierarchy]` (config.toml:24) and baked schema (validation/defaults.rs:192) through the existing validation layer; no new type list.
  - Gate-run readability reuses the full stdout/stderr already stored on `GateRunResult` (types.rs:1072,1074) and the run-list trait `list_gate_runs_for_issue` (storage/mod.rs:230, impls memory.rs:236 and gate_runs.rs); REQ-08 extends the existing `Check`/`CheckAll` variants (cli.rs:971-981) rather than adding a new command (D11). The session-mining sketch (dev/studies/session-mining-jit-improvements.md:227) recommended newest-first run history; we realize that history on `gate check` instead of a separate `gate runs` command.
  - Per-issue validation reuses the existing `run_rules(Some(id))` single-issue path (commands/validate.rs:944-982) and the injected `JIT_ISSUE_ID` (gate_execution.rs:53); the container-subtree checker `jit validate --scope <ID>` already exists (cli.rs:232-240) as a sibling deterministic checker.
  - Repo-wide integrity (REQ-13) reuses `run_rules(None)` (main.rs:4771) wrapped as a config-declared `repo-validate` auto-gate in the registry/presets layer (gate_presets, templates.toml), attached to containers by extending the `plan` template so the `container` anchor requires it; container gate state rides the existing `Issue.gates_status` (types.rs:395). No engine domain literal.
  - `gate update` reuses the atomic write primitive `write_file_atomic` (storage/atomic_write.rs:34) via `save_gate_registry` (storage/json.rs:633→827); per-issue status on `Issue.gates_status` (types.rs:395) is untouched by registry edits, so status preservation is structural.
  - `short_id` injection follows the existing custom-serialize pattern used by `MinimalIssue` (types.rs:544) and the output structs (output.rs:674,830,1028).

- **Layer boundaries:** New flags and nouns stay in cli.rs; orchestration stays in main.rs / commands/*; validation stays in commands/validate.rs and the validation layer; storage edits go through the `IssueStore`/registry-save path and the atomic-write primitive. No new file I/O bypasses storage. Gate-prompt and skill changes (REQ-09, REQ-10, REQ-18) are documentation/script edits with no engine coupling (D8).
