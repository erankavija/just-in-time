Problem · 3 / 15

The bracket was hardcoded — and split in two

The earlier plan-before-fan-out bracket (epic 2fbd2a82) worked, but its assembly was asymmetric:

  • jit plan created only P; the breakdown node B was hand-assembled by the jit-breakdown skill from primitives — wiring in skill prose, not the binary.
  • jit plan left P's description empty, weakening the plan-review gate, whose prompt reads issue.description for criteria.
  • The [planning] config block was effectively a hardcoded two-node bracket declaration.
One specific shape, baked into engine logic and split across two layers — no room to express any other recurring subgraph.
Idea · 4 / 15

A graph template is a reusable subgraph

Replace the one hardcoded bracket with a generic primitive. A graph template is a named, parameterized subgraph the engine instantiates and splices onto existing nodes:

templatedeclared in config ＋anchor bindings Ctarget issue →jit apply subgraphspliced into the DAG
The planning bracket becomes the first template (plan). The mechanism is generic and domain-agnostic: nothing about plan, P, or B is hardcoded in engine logic.
Model · 5 / 15

The model: four ingredients

  • Nodes — each with a role, a type (from the type hierarchy), gates (presets), and doc + description templates.
  • Internal edges — role-based depends_on (e.g. breakdown depends on planning).
  • Named anchors — roles (e.g. container) bound to existing issues at apply time; nodes wire to anchors.
  • Transforms — a small, fixed, DAG-preserving vocabulary of edge rewrites on anchors.
Static, no deferred edges: a template fully instantiates at apply. Later structural changes (the breakdown interior splice) happen independently.
Model · 6 / 15

Declared in .jit/templates.toml

[[template]]
name = "plan"
[[template.anchors]]
name = "container"      # bound at apply to the target
[[template.nodes]]
role = "planning"
type = "planning"       # must exist in [type_hierarchy]
gates = ["plan-review"] # gate-preset names
The template declares applies_to = ["epic"] and a second breakdown node (two gates, a B → P edge). The loader validates roles, edges, types, and acyclicity at every config load.
Command · 7 / 15

One command, one operation

jit apply plan <container-C>

Applying plan to a container C does, atomically:

  • create P (type:planning, plan-review) and B (type:breakdown, coverage-preview + breakdown-review) with fresh ids;
  • wire internal edge B → P and anchor edge C → B;
  • apply the gate presets and interpolate descriptions;
  • run the move-upstream-to-planning transform.
This replaces both the old jit plan (P-only) and the skill's hand-wiring of B. jit plan / --with-planning are gone.
Result · 8 / 15

The plan template reproduces the bracket

Ccontainer →depends on impl…the work →depends on Bbreakdown · coverage →depends on Pplanning · plan-review
Precedence P > B > impl > C: plan first → breakdown → the work → the container closes.
Identical spine to epic 2fbd2a82 — but now P and B are both born from config in a single jit apply, and the result is always re-validated as an acyclic DAG.
Mechanism · 9 / 15

Interpolation closes the empty-P gap

description and doc fields interpolate from anchor context — container title, id, short id, [hard] criteria, labels, resolved doc path:

doc         = "dev/active/{container.id}-plan.md"
description = "Planning node for {container.title} ({container.short_id}).
  Produce the plan that decomposes this container before fan-out:
  address its [hard] success criteria… Plan document: {doc}."
Applying plan now yields a planning node with a populated description, so plan-review reads real criteria. REQ-03 closed; the two legacy empty-P descriptions (1eb0bdfd, 2937919f) were reseeded to the exact template literal.
Mechanism · 10 / 15

Transforms: small, fixed, DAG-preserving

A template may rewrite edges on its anchors through a fixed vocabulary with an extensible dispatch. This epic ships exactly one:

move-upstream-to-role  ·  as move-upstream-to-planning
  • Moves the container C's pre-existing upstream dependencies onto P, so planning waits on upstream work and C becomes the pure closure node.
  • Reproduces the retrofit behaviour the old jit plan did by hand — now declarative.
Other transforms (e.g. redirect) are deferred: the engine enables them, but only move-upstream-to-role ships. Every transform must keep the graph acyclic.
Mechanism · 11 / 15

Apply is atomic and cycle-safe

validate → snapshot → instantiate → re-validate
  • The whole apply is one operation: nodes, internal + anchor edges, gate presets, transforms — then the result is validated as an acyclic DAG.
  • A cycle-atomicity gap surfaced during the epic: apply could partially mutate before a cycle was detected.
  • Fixed with an edge-simulation pre-check — the resulting edge set is simulated for cycles before any node is created.
Tested entirely against isolated temporary jit repositories, never the production .jit store (REQ-08).
Integration · 12 / 15

Breakdown consumes the pre-created B

With B now born at jit apply, the breakdown engine and the jit-breakdown skill no longer create it — they perform only the interior splice:

sourcesentry impl →depends on Bpre-created ┊ Ccontainer →depends on sinksfinal impl
B-creation removed from skill prose (REQ-07). The skill drafts children, wires sources → B and C → sinks, and runs the coverage gate — no hand-assembly of the bracket.

Delivery · 14 / 15

Dogfood surfaced three jit gaps — fixed in-epic

Clean-migrating a real legacy container (25064508) to jit apply plan exercised delete + dep-rm on the production store and exposed pre-existing bugs:

  • jit issue delete logged no event — no IssueDeleted type existed.
  • jit dep add/rm logged no issue_updated; no-op removals bumped updated_at without an event.
  • jit apply plan detected only an existing breakdown node, so it could duplicate the planning node on a legacy P-only container.
All three fixed within the epic, not deferred. Only the aggregate dogfood — not the per-task reviews — caught them; story-level code-review earned its keep.
Wrap-up · 15 / 15

Summary & what it opens up

  • The hardcoded planning bracket is now one configuration of a generic graph-template engine.
  • jit apply plan creates P + B from config in a single, cycle-safe, atomic operation — closing the empty-P gap and the binary/skill split.
  • The flat [planning] / PlanningConfig path is gone; nothing domain-specific is hardcoded in engine logic.
  • Next: new recurring subgraphs become config, not code — additional templates and transforms need no further engine work.

The bracket is now a template.
