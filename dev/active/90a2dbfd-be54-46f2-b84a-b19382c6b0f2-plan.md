## 2. Technical soundness and architectural fit

- **Approach:** Treat "item kind" as fully data-driven over two source substrates ‚Äî markdown
  for descriptive kinds, toml for kinds with typed behaviour-driving fields ‚Äî and remove the
  three places where a domain concept (`invariant`) or a baked kind name leaks into engine or
  pure code. The markdown substrate is already generic; the work is to (a) prove it by
  contract, (b) generalize the toml substrate the same way, (c) delete the now-dead reserved
  name and baked defaults, and (d) two configuration-only deliverables (drift simplification,
  CLAUDE.md region projection).

- **Reuses / integrates with:**
  - `ItemKind` generic 9-field tuple and `from_config` builder ‚Äî `crates/jit/src/domain/item.rs:425-435`, `:462`.
  - `resolve_item_kinds` (None ‚áí built-ins, Some(map) ‚áí verbatim, **replacing** built-ins) ‚Äî `crates/jit/src/domain/item.rs:1229-1243`.
  - Markdown project path: `project_items` ‚Üí `storage.read_repo_file` ‚Üí `index_project_sources` ‚Äî `crates/jit/src/commands/item.rs:213-242`; path-validated repo-local read at `crates/jit/src/storage/json.rs:762` (its sibling `write_repo_file` at `:792`).
  - The REQ-02 seam: `index_project_sources` parses each `ProjectSource` via `extract_raw_items`, then `raw.extend(registry_items)`, then one `derive_scope_items(&Scope::Project, raw)` ‚Äî `crates/jit/src/domain/item.rs:1155-1169`.
  - `SourceOfTruth { MarkdownFirst, RegistryFirst }`, default MarkdownFirst ‚Äî `crates/jit/src/config.rs:547`; routing on `kind_scope().is_project()` then `match source_of_truth()` ‚Äî `crates/jit/src/commands/item.rs:193-234`.

- **Architecture-boundary hazards (called out per constraint):**
  - The item-kind engine must hardcode no domain concept. The exact violations are the live
    `name=="invariant"` branches (`crates/jit/src/domain/item.rs:496`,
    `crates/jit/src/commands/item.rs:202`) and the baked `*_default` kind-name constants. REQ-03/REQ-04
    remove them. **Hazard:** the removal must route registry selection through the config
    `source` descriptor ‚Äî it must not relocate an `"invariant"` literal into pure/engine code

    deletion safe.
  - All filesystem IO stays at the storage boundary. The REQ-02 toml read goes through
    `read_repo_file` (as the markdown path already does at `crates/jit/src/storage/json.rs:792`),
    and the REQ-06 `CLAUDE.md` write goes through the existing storage-backed projection
    write-back (`write_repo_file`, `crates/jit/src/storage/json.rs:792`), not `std::fs`.
  - Every new public fn/struct (the toml-source loader, the `DriftFinding` subject-kind, any
    descriptor type) carries the code-review gate cost: `# Examples` doc, typed `thiserror`
    errors, atomic writes. Budgeted in ¬ß4 actionability.

## Decisions

- **D1 ‚Äî Source-substrate split:** chosen **markdown-first for descriptive kinds
  (decision/definition/risk), registry-first(.toml) only for kinds with typed
  behaviour-driving fields (invariant: kind/enforced-by)**. Rejected: everything-toml (forces
  field-mapping even for prose; heavier), everything-markdown (cannot carry typed fields).
- **D2 ‚Äî Generalize the registry seam:** chosen a **config `source` descriptor (toml path +
  table + id-field + text-field + link-fields) threaded into `index_project_sources`, keeping
  `InvariantRegistry` as the typed validator**. Rejected: a full generic schema-mapping
  language (over-engineered), perpetuating per-kind hardcoded loaders (the current smell).
- **D3 ‚Äî Retire the reserved name:** chosen to **drop the reserved name and
  `InvariantMustBeProjectRegistryFirst`, routing registry selection through the source
  descriptor, not a relocated literal**. Rejected: keep the reserved name (perpetuates the
  domain leak `CLAUDE.md` forbids).
- **D4 ‚Äî Demote baked defaults:** chosen a **jit-init-authored `[item_kinds]` table; the
  binary carries no domain kind names; migrate this repo's config in the same change**.
  Rejected: keep `builtin_default_kinds` (mild domain-agnostic violation). Single real
  consumer + this repo ‚áí no backward-compat needed.
- **D5 ‚Äî Drift simplification:** chosen to **remove the enforced-but-undeclared direction
  entirely (no rule or gate subject reported), leaving declared-but-unenforced as the sole
  error**. The seed's 18 findings are 11 gate + 7 rule subjects; dropping only the gate half
  would still leave 7 rule findings and a non-zero `jit invariant check`, so a gate-only or
  subject-kind exclusion cannot meet REQ-05's "exits 0 on the seed". Rejected: keep nagging
  every rule/gate (no value ‚Äî owner decision); subject-kind/severity-tier bucketing (more
  machinery, and insufficient for exit-0 on its own). Rules become self-accounting only once
  they are addressable items (REQ-07, deferred), so until then the direction has no payload.
- **D6 ‚Äî CLAUDE.md region projection:** chosen to **insert begin/end markers into CLAUDE.md's
  "Domain Invariants" section and set `[invariant_projection]` `target=CLAUDE.md`,
  `mode=region`, reusing the existing region projection**. Rejected: separate-file-only (leaves
  the duplication the issue targets).
- **D7 ‚Äî REQ-07 deferral:** chosen to **defer rule/gate-as-item**; blocked by the
  `labels.rs:24` value-grammar `:` collision with rule self-ids like
  `default:namespace-unique:type`. Out of the hard floor.
- **Assumptions:** the existing `docs/reference/glossary.md` is an acceptable markdown source
  for the dogfood `definition` kind authored in REQ-04's table (Group C) (it exists and is
  descriptive prose) ‚Äî rationale: reuses existing content rather than adding new; risk if
  wrong: a different markdown source must be named, a config-only adjustment. The
  `### Domain Invariants` section of `CLAUDE.md` (line 123) is the intended projection region ‚Äî
  rationale: it is the duplication the registry should own; risk if wrong: markers move to the
  correct section, marker-only edit.
