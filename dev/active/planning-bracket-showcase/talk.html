Problem · 3 / 14

One container, three conflated jobs

Before this work, a container (an epic, or whatever a ruleset's top type is) silently played three roles at once:

  1. the planning artifact — a design doc produced out-of-band;
  2. the fan-out trigger — breakdown, run speculatively;
  3. the fan-in acceptance node — coverage checked at → done.
Breakdown happens before anyone reviews whether the decomposition is right, and coverage is only checked at the very end — after all the work is done.
Solution · 4 / 14

The bracket: a reviewed, gated spine

A breakable container C is bracketed by two function-typed nodes — P (type:planning) and B (type:breakdown) — with the implementation subgraph spliced between them:

Ccontainer →depends on impl…the work →depends on Bbreakdown · coverage gate →depends on Pplanning · plan-review gate
Read as precedence P > B > impl > C: plan first → breakdown → the work → the container closes.
Solution · 5 / 14

Edge geometry (kept transitively reduced)

  • C → impl: sinks only. Only the issues immediately before container completion carry an edge to C.
  • impl → B: sources only. Only entry impl issues depend on B; this transitively gates all impl behind the approved breakdown.
  • B → P. Breakdown depends on the plan; B is born at breakdown, not at scaffold.
  • Reduction drops C → P automatically once the spine connects C → impl → B → P.
Retrofit: jit plan <existing-C> moves C's pre-existing upstream deps onto P, so planning waits on upstream work and C becomes the pure closure node.
Gates · 6 / 14

Two gates, two questions

plan-review · on P

An agent quality gate. Is the plan good enough to fan out from? Blocks breakdown until the plan is approved.

coverage-preview · on B

A deterministic gate: runs jit validate --scope <C>. Do the drafted children actually cover C's [hard] criteria? Exit 4 if not.

Quality vs coverage are split across the two nodes. The gate runs through the standard runner — the breakdown step attaches the gate; it never fakes a result.
Gates · 7 / 14

Preview vs closure: one rule, one knob

Both checks are instances of the same label-coverage rule. They differ only by the child-state knob:

aspectpreview (on B)closure (on C → done)

The literal "any" is not a token — an absent child-state already means "any state" in the evaluator. No parser change was needed.
Engine · 8 / 14

Why coverage had to go transitive

Because the DAG is reduced, C links only to its sink children — so a criterion satisfied by a non-sink impl issue read as uncovered. Three engine changes fixed it:

  • Transitive traversal by default — label-coverage walks the dependency closure, not just direct adjacency.
  • child-type-exclude — drops bracket types from candidates and halts the walk at them, bounding coverage to the impl interior between C and B.
  • container-from-label — a rule firing on B resolves its container from the brackets:<C-id> label (full id, short id, or prefix).
The per-issue rules were left untouched — only label-coverage became transitive.

Delivery · 10 / 14

Domain-agnostic by construction

The engine bakes in no domain vocabulary. Two example rulesets prove it, declaring their own breakable + bracket types in config:

SDD

epic is breakable; criteria are REQ-* requirements satisfied by impl children.

Research

goal is breakable (no epic); [hard] hypotheses tested by experiments.

Verified repeatedly: no epic / planning / breakdown string literals in production Rust — every type name flows from config (the flat planning-config block then, the .jit/templates.toml graph template now).
Delivery · 11 / 14

Steering eval: three scenarios

Plan rejected

The plan-review gate blocks a deficient plan before any fan-out.

Coverage gap

An uncovered [hard] criterion is caught at the breakdown gate — exit 4.

Green path

A well-formed bracket validates clean and passes end-to-end.

Added as data-only fixtures (scenario.toml) to the existing deterministic steering harness — no Rust changes; the runner auto-discovers them.
Delivery · 12 / 14

The skills now drive the order

plan → review → breakdown → coverage → implement
  • jit-breakdown — for a breakable container, builds the spine: creates B, drafts children in Backlog with satisfies: labels, wires sources→B / C→sinks, then runs the coverage gate via the standard runner.
  • project-lead — scaffolds P, blocks breakdown until plan-review passes, blocks the implementation fan-out until coverage passes. Breakable types read from the ruleset, never hardcoded.
Wrap-up · 13 / 14

Summary

  • Planning is now a reviewed, gated node in the DAG — not an out-of-band step.
  • Coverage is checked before fan-out (preview on B) and again at closure (on C) — one rule, one knob.
  • The engine stays domain-agnostic; SDD and research rulesets prove it.
  • The whole flow is regression-tested by the steering suite.
