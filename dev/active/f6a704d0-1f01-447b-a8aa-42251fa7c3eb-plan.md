# Plan — Generic issue & label validation engine (JSON-Schema-based), with SDD as first application

Container: `f6a704d0` (epic). State: `done`.

## Status

Retroactive plan stub. This epic predates the planning-bracket mechanism and was completed without an external plan document; this stub exists so `jit validate` (which requires a plan doc for every breakable container type) passes repo-wide. It is a historical record, not an active plan.

## Success Criteria (recorded from the epic description)

- [hard] Rules load from `.jit/rules.toml`; schemas inline or from `.jit/schemas/`; the loader rejects mixing shorthand and raw schema in one rule.
- [hard] Selector matrix (type, label, state, doc-type) matches issues; applicable rules union.
- [hard] Local rules evaluate on create/update; per-rule `enforce` (default false) blocks when set; `--force` bypasses.
- [hard] Graph rules (coverage, reference-integrity, dependency-shape) run only in `jit validate`/gate checkers.
- [hard] Validation runs against the enriched projection; Markdown in the default build, HTML/XML behind optional features, all behind one parser trait.
- [hard] JSON Schema validation via the `jsonschema` crate; shorthand kinds desugar to JSON Schema.
- [hard] The COMPLETE set of existing checks (validation.rs + validate.rs + `[validation]` + `[namespaces]` enforcement) is re-expressed as default rules with preserved behavior; legacy enforcement keys are migrated and removed (no backward-compat aliases).
- [hard] `jit init` scaffolds defaults; existing repos migrate without silently losing checks.
- [hard] `jit validate [<id>] [--json] [--explain]` reports rule name, severity, message; gates invoke it via `$JIT_ISSUE_ID`.
- [hard] SDD ships as example rules + schemas + docs; description `[hard]`/`[aspirational]` criteria are canonical and labels derived; >=1 non-SDD example included.
- [hard] `cargo clippy --workspace --all-targets` and `cargo fmt` clean; tests cover engine, projection, parsers, desugaring, migration.
- [aspirational] The `x-jit-*` custom-keyword extension point is demonstrated.
