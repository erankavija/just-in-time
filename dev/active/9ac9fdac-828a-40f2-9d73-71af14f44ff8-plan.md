# Plan — Graph templates: configurable parameterized subgraphs applied to the work DAG

Container: `9ac9fdac` (epic). Planning node: `7d88e37d`. This plan is the spec the breakdown consumes; it is written to pass `plan-review` and to be decomposable.

## Goal

Generalize the hardcoded planning bracket into a generic **graph-template** layer: named subgraph templates declared in `.jit/templates.toml`, instantiated by `jit apply <template> <container>`. The `plan` template is the first (and, this epic, only) template; the mechanism is generic and domain-agnostic. `apply plan` creates **P+B** in one operation with seeded descriptions; breakdown consumes the pre-created B.

## Design decisions (from the planning interview)

- **Unify on a generic core.** Build `apply_template` as the engine; reimplement P-scaffolding on it and **split** `bracket_breakdown` so B-creation moves into `apply plan`; breakdown consumes a pre-created B.
- **`plan` lives fully in `.jit/templates.toml`** with `applies_to = ["epic"]`; the `[planning]` block + `PlanningConfig` are **removed once `plan.rs` and `breakdown.rs` no longer reference them** (deferred to W4, not W1 — see waves). Only **this repo's** config is rewritten; **gf2 has no `[planning]` block** (it never adopted the bracket) so it needs no change.
- **Remove `jit plan` and `jit issue create --with-planning`**; `jit apply plan` replaces them. Migrate the skills (`project-lead`, `jit-breakdown`) and the bracket-command memory in this epic.
- **CLI:** `jit apply <template> <container> [--anchor role=id]… [--force] [--json]`. Positional container for the common single-anchor case; `--force` re-applies.
- **Transforms:** ship only `move-upstream-to-role` via an extensible dispatch; `redirect` deferred.
- **Breakdown precondition:** if a breakable container has no B, breakdown **errors** and points to `jit apply plan <C>` (never creates B itself).
- **Atomicity:** validate everything before the first mutation; a failure leaves zero partial nodes.
- **Backfill:** manual (not engine) — reseed `1eb0bdfd`/`2937919f` by hand; `7d88e37d`/`59118ee9` are already seeded.
- **Tests:** unit + integration (temp repos) + proptest invariants + post-completion dogfood.

## templates.toml schema (concrete)

```toml
[[template]]
name        = "plan"
description = "Plan-before-fan-out bracket: planning node P and breakdown node B."
applies_to  = ["epic"]           # container types this template may be applied to

  [[template.anchors]]
  name = "container"             # bound at apply time to the target issue

  [[template.nodes]]
  role        = "planning"
  type        = "planning"       # must exist in [type_hierarchy]
  gates       = ["plan-review"]  # gate-preset names (apply_gate_preset)
  doc         = "dev/active/{container.id}-plan.md"
  description = "Planning node for {container.title} ({container.short_id}). Produce the plan that decomposes this container before fan-out: address its [hard] success criteria and make it concrete enough to break into well-ordered work. Plan document: {doc}."

  [[template.nodes]]
  role        = "breakdown"
  type        = "breakdown"
  gates       = ["coverage-preview", "breakdown-review"]
  labels      = ["brackets:{container.short_id}"]
  description = "Breakdown node bracketing {container.title} ({container.short_id}). Holds the decomposition of the approved plan into well-ordered child issues that cover the container's [hard] success criteria."
  depends_on  = ["planning"]     # internal edge: B → P

  # Edge to an anchor: container depends on the breakdown node (C → B)
  [[template.anchor_edges]]
  from = "container"             # anchor role
  to   = "breakdown"            # node role
  # direction is "anchor depends on node"

  [[template.transforms]]
  kind = "move-upstream-to-role"
  role = "planning"             # move container's pre-existing upstream deps onto P
```

Notes: nodes inherit the container's non-`type:` membership labels (as `bracket_container` does today). `gates` are **preset names** resolved through the existing preset manager. Interpolation context: `{container.id|short_id|title}` and `{container.hard_criteria}`; `{doc}` resolves to this node's `doc`. Loader validates at `JitConfig::load`: roles unique; `depends_on`/`anchor_edges`/`transforms` reference declared roles/anchors; node `type`s exist in `[type_hierarchy]`; gate presets exist; internal edges acyclic.

## The apply engine

`commands/template.rs::apply_template(template_name, container_id, anchor_bindings, force) -> TemplateApplyResult`:

1. **Validate before mutating** (mirror existing engines): template exists; container type ∈ `applies_to`; all anchors bound and resolvable; not already applied unless `--force`; simulate edges for acyclicity. Abort clean on any failure.
2. **Instantiate nodes** in template order: `create_issue` with `type:<role-type>`, inherited membership labels, template `labels` (interpolated), interpolated `description`; set `doc` via the `{…}`-extended interpolation + `DocumentReference`; apply each node's gate presets via `apply_gate_preset`.
3. **Wire edges:** internal `depends_on` (by role) and `anchor_edges` (anchor↔node) via `add_dependency` (gets cycle-check + transitive reduction for free).
4. **Run transforms:** `move-upstream-to-role` moves the container's pre-existing deps onto the bound role's node (the existing `bracket_container` logic, generalized). Dispatch is an enum → extensible.
5. Return `TemplateApplyResult { template, anchor_bindings, created_node_ids_by_role }`.

`jit plan`'s old behavior = `apply plan` with the built-in `plan` template. `--force` re-applies (detect existing nodes by role/`brackets:` label; refresh rather than duplicate) — also the basis for any future reseed.

## Risks & mitigations

- **Removing the flat config too early breaks the build.** `PlanningConfig` is consumed by **both** `plan.rs` and `breakdown.rs`. Mitigation: W1 is purely additive; the `[planning]`/`PlanningConfig` removal is deferred to **W4**, after `plan.rs` (W3) and `breakdown.rs` (W4) stop referencing it — so every wave compiles.
- **Splitting `bracket_breakdown` mid-flight breaks the existing bracket workflow.** Mitigation: land W2/W3 (apply makes B) before W4 (breakdown stops making B); keep the bracket green via tests at each step.
- **Removing `jit plan`/`create --with-planning` strands callers.** Mitigation: W3 deletes the command, its doc-comments (cli.rs:51-64), and `tests/plan_scaffold_tests.rs`; W5 migrates `project-lead`, `jit-breakdown`, the memory, and the user docs (`adopt-planning-bracket.md`, `planning-bracket.md`). Acceptance = clean `grep -rE 'jit plan|--with-planning'` over repo + skills.
- **gf2 is not a rewrite.** gf2 has no `[planning]` block, no planning/breakdown types, no `type:planning` issues — it never adopted the bracket, so there is nothing to migrate. W5 only documents the opt-in path. (The earlier "rewrite both configs" framing was wrong; REQ-10 corrected.)
- **Description/doc interpolation is net-new code.** Mitigation: simple `{token}` substitution over a fixed context map (`{container.id|short_id|title|hard_criteria}`, `{doc}`), seeded by `plan_doc.rs`'s `{id}` `String::replace` — not a templating language.
- **`--force` re-apply must update, not duplicate.** Mitigation: detect existing role nodes via the `brackets:` label / role; covered by a proptest (idempotent under `--force`).
- **Open question:** does any rule require breakable containers to *be* bracketed (so `applies_to` needs enforcement)? This repo's `rules.toml` has only default rules, so no enforcement is added now; `applies_to` is an apply-time guard only.
