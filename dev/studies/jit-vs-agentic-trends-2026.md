# jit's New Developments vs. Existing & Emerging Trends in Agentic Coding

## 1. Bottom line

jit is **ahead of the curve on coordination mechanism and behind it on the agent experience layer**, and that split defines its strategic position. Its DAG enforcement, claim/lease system, and exit-4 transition gating are genuinely the most complete in the comparison set (BMAD, cc-sdd, GSD, OpenSpec, Ruflo, superpowers) — every competitor enforces plan-before-code and quality gates through prompt pressure that an agent can silently skip, while jit enforces them at a `apply_state_transition` chokepoint that returns a machine-readable code no prompt-driven framework can replicate without becoming jit. But the field's empirical bottleneck in 2025-2026 is not DAG correctness; it is keeping a single agent coherent *within* a long trajectory (context rot, plan-drift). Cross-session memory is the half jit already **is**: a git-versioned, structured, addressable store of issues, plan docs, decisions and events is durable project memory by construction, and a more deterministic one than the vector stores (Mem0, Cognee, Ruflo's AgentDB) the field is bolting on. jit's real gap is narrower and on the *activation* side — it holds that state but does not push it back into a running agent (no plan-reinsertion), and recall over it is lexical, not semantic. jit owns the bookkeeping layer; every competitor owns the workflow the user actually touches, and jit has one real consumer (gf2) plus itself, with no planning skill yet populating its flagship P node. Of the six developments, two shipped features (lifecycle-aware validation, the generic validation engine) are solid and on-trend, the plan-before-fan-out bracket is a principled architectural lead, and the two largest in-flight epics (graph templates, structured knowledge) are speculative generalizations for an N=1 or future-kinds reality. The highest-ROI work — agent-CLI ergonomics, mined from 209 real session failures — is sitting in backlog behind those two speculative epics. Net: jit is upstream of demand, building an excellent backend before the front end that creates rows in it exists.

## 2. The field in mid-2026

Across the four lenses the field is converging on a layered model with one dominant anxiety underneath it.

**Spec-driven development** has consolidated into a Spec → Plan → Tasks → Implement discipline, productized by **GitHub Spec Kit** (Sept 2025, agent-agnostic, lightweight) and **AWS Kiro** (July 2025, opinionated IDE with `requirements.md`/`design.md`/`tasks.md` in EARS notation, Hooks, Steering). **BMAD-METHOD** pioneered the persona-driven multi-phase pipeline that both productized. The honest caveat the marketing buries: **living/bidirectional spec-code sync is aspirational, not delivered** — Kiro itself calls drift "requires discipline," and McAree's critique found the sync claim oversold.

**Planning and orchestration** has converged from egalitarian multi-agent (which Cursor found degrades to 2-3x throughput at 20 agents due to lock contention) onto a **hierarchical Planner/Worker/Judge triad**. **Plan/Act separation is now harness-enforced** (Claude Code disables Edit/Write in plan mode; Cursor's named toggle). The frontier has explicitly moved from generation to **verification**: dedicated reviewer agents firing at task boundaries, motivated by DORA 2025's finding that 90% AI adoption correlates with +154% PR size and +9% bug rate.

**Context engineering** is the unifying threat the per-development view never names. `AGENTS.md`/`CLAUDE.md` standardized under the Linux Foundation AAIF, but **context rot** (Anthropic, Sept 2025) — coherence degrading non-linearly with session length — is the recognized failure mode. The countermeasures the whole field is racing on: **Skills/SKILL.md** for just-in-time loading, tiered hot/warm/cold memory, **Liu et al.'s plan-reinsertion** finding (compliance decays over 16,991 trajectories; reinject every ~5 steps), cross-session memory frameworks (**Mem0, Cognee, Ruflo's AgentDB**), and CI-integrated drift detection. Gloaguen et al. (ETH/DeepMind) deliver the cold counterweight: **more config hurts** — context files reduced SWE-Bench success and raised cost >20%; models reliably follow ~150-200 instructions and every line competes.

**Agent-tool interfaces**: MCP won transport; **CLI won local orchestration** (10-32x cheaper than MCP for local tools per the scalekit benchmark). The research edge is **Normalized Tool Contracts** (arxiv 2605.10555: confidence/evidence/next-action fields, ID hallucination 28%→4%) and **deterministic gates with exit-code verdicts**. jit sits squarely inside the **repo-local tracker wave** (kata, git-bug, **Beads** by Steve Yegge — 11.1k stars, the real heavyweight peer — Tracer, GNAP), explicitly named in the trend corpus.

jit is the only tool in this set that lives natively in the *intersection* of "repo-local CLI tracker" and "deterministic gates" — but it is absent from the SDD-experience layer, and within the context/memory layer it *owns* durable cross-session state while ceding only the within-trajectory activation half (re-injection, semantic recall) where much of the field's remaining energy sits.

## 4. Where jit is genuinely ahead (survived adversarial review)

These are the differentiators a smart skeptic at a competing project would concede and could not retrofit:

- **DAG enforcement as a first-class tracker primitive.** Transitive reduction + cycle detection in `commands/dependency.rs` before every write, blocking-analysis at the command layer. Every peer has either no dependency model (BMAD, OpenSpec, superpowers), text annotations only (cc-sdd `_Depends:_`, GSD `ROADMAP.md` ordering), or ephemeral records (Ruflo). Beads is the only structural peer and still has no published exit-code/gate contract.
- **The claim/lease system.** File-locking with explicit lease records, force-evict, renewal, and conflict detection. It is the **only mechanism in the corpus that survives concurrent agents** rather than segregating them (GSD uses git-worktrees; superpowers has nothing). This is structurally unique.
- **Plan-before-fan-out as a DAG invariant.** `query_ready` returns nothing from the impl subgraph until P's plan-review gate and B's coverage-preview gate clear. This is mechanically stronger than Claude Code's plan-mode toggle or Kiro's human checkboxes — and once you have watched an agent silently skip a prompt-enforced gate, a binary returning exit 4 is the only adult in the room.
- **The `apply_state_transition` chokepoint + exit-code taxonomy.** A single enforced path with `ValidationFailed=4` distinct from infra/not-found/generic, plus `TransitionBlocked`/`GraphRuleBypassed` audit events. Prompt-driven frameworks (cc-sdd, GSD, superpowers) produce verdicts the *same model* can rationalize around; jit's is interpreter-independent and the bypass is logged.
- **The two-instance coverage rule (preview vs closure from one rule kind by toggling `child-state`)** and **transitive coverage with type-bounded halting** (`child-type-exclude`). Economy of mechanism no studied framework has; the four revision cycles in the design doc confirm it was genuinely subtle rather than trivial.
- **Domain-agnosticism proven by two complete, non-software rulesets** (the `research/` example uses `goal`/`experiment`/`hyp:` vocabulary with zero software strings). BMAD, Kiro, Spec Kit, cc-sdd are all hardwired to software feature development.
- **Methodological rigor: the pre-decomposition verification pass** in the ergonomics plan (rebuild the binary, separate already-shipped from still-broken before decomposing) prevented ~6-8 ghost work items. No research framework in the corpus documents this step.

## 5. Where jit lags or is blind (real gaps, not non-goals)

- **Within-trajectory *activation*, not cross-session memory, is the real gap.** jit **is** a cross-session memory system — a git-versioned, structured, addressable store of issues, plan docs, decisions/risks/invariants plus an event log is exactly the durable project memory Mem0/Cognee/AgentDB approximate with vector stores, except jit's is deterministic and git-auditable rather than fuzzy. What jit lacks is the *push* side: once P's gate passes, nothing re-injects the plan into context mid-implementation (no plan-reinsertion, the intervention Liu et al. quantify), and recall over the store is lexical (`jit search`) not semantic. cc-sdd propagating Implementation Notes forward and superpowers' compaction-safe progress ledger are *activation* mechanisms layered over memory jit already persists more rigorously than either. The bracket solves pre-fan-out plan *theater*; the field's evidence points at mid-trajectory *drift* — which jit has the substrate to fix but does not yet act on. (`events.jsonl` is the audit/episodic layer; the issues, plan docs and structured items are the queryable memory, and the structured-knowledge epic is jit *maturing* that layer, not a sign it is missing.)
- **No planning skill populating the P node.** Issue eed6750c is "ready," no `SKILL.md` exists. BMAD, cc-sdd, GSD, and superpowers all ship their planning workflow as the *primary product*. jit can gate on plan quality but cannot produce or validate plan content — the flagship bracket is a gated empty box.
- **No opinion on the autonomous execution loop.** cc-sdd ships an implementer/reviewer/debugger trio with bounded retries; GSD ships wave-based execution with worktrees; superpowers ships sequential dispatch with a resume ledger. jit enforces gates around a claimed issue but says nothing about how work gets done inside it. The field's measured bottleneck is execution-loop quality, which jit cedes entirely.
- **No item→item cross-reference graph.** The structured-knowledge epic explicitly defers risk→decision→requirement links to a future layer. Kiro (`design.md↔requirements.md`) and BMAD (epics→FR-N/NFR-N) handle this *today* in prose.
- **Non-blocking graph findings are returned as string warnings, not a machine-parseable field.** An agent consuming `--json` from `issue update` cannot reliably branch on them — a real gap in the very machine-readable story jit sells.
- **No stable finding codes / next-action hints.** The NTC pattern (33% fewer calls, ID hallucination 28%→4%) puts next-action and evidence in structured fields; jit's remediation is prose ("`jit issue update <issue> # satisfy or fix rule`"), names the criterion but not which child needs the `satisfies:` label, and cross-run dedup requires message parsing.
- **`checker-command` is described as a feature but never executes** — a silent correctness failure for any ruleset that relies on it.
- **Token economics untouched.** Verbose `--json` field names, internal `gates_required`/`gates_status` diverging from the public contract across ~356 source sites with the fix scoped to `output.rs` only. TOON/flat-schema concerns are live in the field.

## 6. Strategic risks

- **Upstream of demand (the dominant strategic risk).** jit ships the bookkeeping the experience would need *if the experience existed*. One real consumer (gf2), no planning skill, no execution loop. The risk is not being wrong about DAGs — the trend data validates repo-local CLI trackers — it is being a beautifully-engineered backend for a front end nobody has built. Beads (11.1k stars, shipping a Gas Town *orchestration layer* on top) is the cautionary peer: it pairs the tracker with the experience.
- **Sophistication outrunning adoption, against the field's actual direction.** Graph-templates (N=1), structured-knowledge's five data shapes, and the 14-kind rules DSL all add declarative surface (`templates.toml`, `invariants.toml`, `rules.toml` with 8+ knobs) precisely as Gloaguen et al. show *more config hurts agents* and the field flees toward lean, just-in-time context. A ruleset author who must understand `child-state` absent-vs-`"done"` to dodge the inversion bug **that shipped** is the canonical signal.
- **The lifecycle-inversion bug is a recurring class, not a one-off.** It shipped, survived review, survived the iterations-to-green eval, and was caught only post-deploy. The structured-knowledge epic introduces a second such subtlety (per-kind source-of-truth direction: markdown-first vs registry-first sharing one engine). The LLM-agent eval that should catch this is **not in CI**; a rule change that preserves exit codes but degrades message quality is invisible between manual runs.
- **Priority inversion.** The ergonomics epic — strongest empirical grounding (209 findings), highest ROI (gate-pass bug in 21/42 sessions) — is deprioritized behind two speculative generalization epics. Even within that epic, the plan distributes effort across 11 tasks at equal depth and *dropped* inline checker output (forced as a follow-up command in 18/42 sessions) as "the checker's responsibility," inverting the priority order the session evidence established.
- **Single-consumer over-fitting + breaking changes.** Optimizing for Claude Code's patterns risks breaking other clients; the planned 0/1/2 gate taxonomy breaks the current 0/4 behavior agents already adapted to, and the same broken `gates_required`/`gates_status` shape is exposed to any MCP client.
- **main.rs monolith (4,928 lines).** Adding `jit item *`, `jit invariant *`, and `jit apply` pushes a `#[deny]`-disciplined codebase further toward command handlers embedding logic that belongs in domain/storage.

## 7. Recommendations

**Double down on (the defensible moat):**
1. **The chokepoint + exit-code contract + claim/lease system.** This is the one thing no prompt-driven framework can replicate. Harden it: move the LLM-agent steering eval **into CI** so message-quality regressions surface on PR, and add a semantic-misconfiguration validator (`"did you mean to omit child-state?"`) so the next author does not re-ship the inversion bug.
2. **Ship the agent-CLI ergonomics epic next — reorder it ahead of the two planning epics.** It has the strongest evidence and highest ROI in the portfolio. Within it, **fix the top two mined problems deeply** (gate-pass exit taxonomy, inline checker output) before the 9 long-tail items; reconsider dropping inline output given 18/42 sessions needed it. Fix `gates_required`/`gates_status` at the schema source so MCP propagates automatically.

**Cut or defer (sophistication the market won't pay for):**
3. **Defer graph-templates (9ac9fdac).** Do not ship a transform vocabulary and TOML schema for one template. The minimal fix is an enriched flat planning-config block with description templates plus moving B-creation into the scaffold command. Revisit only when a genuine second template is in hand — and note REQ-10's "both real configs rewritten" is inaccurate, since gf2 has no flat planning-config block at all.
4. **Radically narrow structured-knowledge (25064508).** The defensible nucleus is the **invariant-projection workflow** (one kind, `invariants.toml`, render/check). Drop the generic four-kind addressing model and the `jit item *` surface until a second kind is actually needed. Six design problems in one epic with ten `[hard]` reqs is the over-engineering pattern. Fix the inert `checker-command` (implement with sandboxing, or delete it and the docs that call it a feature) in the same sweep.

**Build the missing half (close the demand gap):**
5. **Write the planning skill (eed6750c) before any further infrastructure.** The P node is empty until it exists. Borrow superpowers' Socratic elicitation and cc-sdd's phase structure. This converts the bracket from formally-correct-empty-box into a shipping experience and is the single highest-leverage move to escape "upstream of demand."
6. **Add a plan-reinsertion mechanism.** This is the field's strongest empirical intervention (Liu et al.) and jit has the perfect substrate: the plan doc is already at a configured path; emit it (or a structured digest) on `claim`/resume and at intervals. Closes the real gap — activating memory jit already holds — with modest effort.

**Watch (let the field decide before investing):**
7. **Semantic recall over the existing store — not a vector-memory product.** jit already *is* the cross-session memory; a Mem0/Cognee/AgentDB-style vector store would largely duplicate what its typed graph + event log do deterministically, and would contradict the "no external database" stance. The only thin gap is fuzzy recall: add an optional semantic index over issues/docs/items *if* `jit search`'s lexical matching proves limiting in practice. Resist building memory jit doesn't need; invest in surfacing the memory it has.
8. **NTC-style structured responses** (next-action/evidence fields) and **token-efficient output** (TOON/flat schemas) — both emerging, not settled. Add stable finding codes and machine-parseable graph-finding fields opportunistically when touching `output.rs` for the ergonomics epic; defer the rest until the standards firm up.
