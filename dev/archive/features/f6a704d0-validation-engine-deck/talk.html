Motivation · 3 / 18

What changed

Before

Validation was hard-coded in Rust and steered by scattered [validation] flags. Adding a check meant editing the binary.

Now

A general, declarative engine. Rules are data in .jit/rules.toml; the same engine serves many methodologies, not just one.

rules.toml is the sole source of truth when present. The legacy enforcement keys were removed — jit is now a flexible substrate for whatever discipline you author.
Concepts · 4 / 18

The mental model

1 · Project

Each issue becomes an enriched projection: its fields, labels, and the parsed sections of its body.

2 · Select

A rule's when selector decides which issues it applies to. Matching rules form a union.

3 · Assert

Each applicable rule asserts a condition over the projection and reports name, severity, message.

One trait parses the body (Markdown by default; HTML/XML behind features), so every rule sees the same shape regardless of format.
Concepts · 5 / 18

Anatomy of a rule

[[rules]]
name    = "epic-has-criteria-section"   # unique identifier, shown in reports
when    = { type = "epic" }             # selector — omit to match every issue
severity = "error"                      # "error" fails validate; "warn" advises
enforce = true                          # error + enforce => also BLOCKS the write
assert  = { require-section = { heading = "Success Criteria" } }
Five fields. assert is the only required one; severity defaults to error, enforce to false, and an absent when matches all issues.

Selectors: the when matrix

keymatches onexample
typethe type:* label value{ type = "epic" }
labelpresence of a label (glob){ label = "req:*" }
statelifecycle state{ state = "in_progress" }
doc-typea linked document's type{ doc-type = "design" }
Keys combine (AND). Omitting when applies the rule to every issue — that is how the canonical default:label-format rule covers all labels.
Concepts · 7 / 18

Severity vs enforce

enforce = falseenforce = true
severity = "error"fails jit validate / gatesfails validate and blocks the write
severity = "warn"advisory warning only(warn never blocks)
severity = how loud (does it fail a run?). enforce = how early (does it stop a create/update at write time?). Use --force to bypass an enforced block, logged.
Concepts · 8 / 18

Local rules vs graph rules

Local

Judge one issue in isolation: format, structure, required sections, schema shape. Evaluate on create/update and can block writes.

Graph

Judge relationships across the DAG: coverage, reference integrity, dependency shape, hierarchy. Run only in jit validate and gate checkers — never on the write path.

Rule of thumb: if answering the question needs another issue, it is a graph rule.

kindchecks

Shorthands are ergonomic sugar — they desugar to JSON Schema under the hood, so behaviour is uniform.
Authoring · 10 / 18

Raw JSON Schema, when sugar is not enough

# Inline, or reference a file under .jit/schemas/
assert = { json-schema = "schemas/spec-body.json" }

Validates against the enriched projection (fields, raw_labels, parsed sections) via the jsonschema crate. Use it for contains, pattern, and structure TOML can't express.

Extension point: custom x-jit-* keywords let you add bespoke checks inside a schema. One rule may use shorthand or raw schema — mixing the two in one rule is rejected at load.

dependency-shapestructural constraints on dependency edges
type-hierarchybuilt-in orphan-leaf / strategic-consistency warnings
scope = "linked" confines resolution to an issue's own dependency graph — so a child cannot accidentally satisfy an unrelated epic's identically-named requirement.
Authoring · 12 / 18

The checker-command escape hatch

[[rules]]
name   = "spec-spellcheck"
when   = { type = "epic" }
assert = { checker-command = { run = "scripts/lint-spec.sh" } }

For checks the engine can't express declaratively, shell out. The command receives $JIT_ISSUE_ID and its exit code / output drive the result.

It is a validate/gate hatch only — a checker-command rule with enforce = true is rejected at load. It must never silently block writes.
Configuring · 13 / 18

config.toml vs rules.toml

config.toml — taxonomy

[type_hierarchy] (the type levels & membership labels) and [namespaces.*] (description, unique, examples). What things are.

rules.toml — enforcement

Every check: formats, allowed values, patterns, coverage, references. What is allowed. Scaffolded from the taxonomy by jit init.

The registry drives default:namespace-registry and default:namespace-unique:*. Allowed-value enums and patterns are no longer config keys — author them as rules.
Configuring · 14 / 18

Format-agnostic bodies: content_format

# repo default in config.toml: [validation] content_format = "markdown"
# override per issue at authoring time:
jit issue create -t "Spec" --content-format html --label type:epic

Selection order: the issue's content_format → the repo default → Markdown. The matching parser feeds the same projected sections to every rule, so HTML/XML specs validate in production, not just tests.

HTML/XML parsers are behind cargo features; selecting a format whose feature isn't compiled is a clear error, never a silent Markdown fallback.
Operate · 15 / 18

Running validation

jit validate                 # whole repo: local + graph rules
jit validate <id> --explain  # which rules matched this issue, and why
jit validate --json          # machine-readable: name, severity, message

Quality gates invoke the same engine per issue via $JIT_ISSUE_ID, so a gate enforces exactly the rules you authored — no separate configuration.

--explain is the debugging workhorse: it lists every rule that matched the issue and whether each passed or failed.

Patterns · 17 / 18

Worked example: spec-driven development

# Coverage: every [hard] criterion in the epic body is implemented by a done child
assert = { label-coverage = { criteria-section = "success_criteria",
  marker = "[hard]", id-pattern = "REQ-[0-9]+",
  satisfies-namespace = "satisfies", child-state = "done", child-link = "dependents" } }

The epic's ## Success Criteria are canonical; req: and satisfies: labels are derived. Local rules check format and structure; graph rules close the derivation loop.

Ships as a copy-in example under docs/examples/sdd/ — methodologies are examples, not built-ins.
Wrap-up · 18 / 18

A getting-started recipe

  1. Run jit init — get a sensible default rules.toml from your taxonomy.
  2. Declare namespaces in config.toml; let init derive the registry/uniqueness rules.
  3. Add your own [[rules]]: start with shorthand, reach for JSON Schema when needed.
  4. Iterate with jit validate <id> --explain; flip enforce = true once stable.
  5. Wire the engine into a gate so it runs on every issue.
Start advisory (warn / enforce=false), tighten to enforced once the rule earns trust.
