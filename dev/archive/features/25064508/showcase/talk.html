Motivation · 3 / 17

Before: a requirement was just prose

  • Regex-parsed. A line in ## Success Criteria matched by a hard-coded pattern — "requirement" was wired into the engine.
  • Locally scoped. REQ-01 was only unambiguous inside one container's subtree — not addressable across the repo.
  • Discoverable only by validating. You found requirements by running the coverage gate, not by querying them.
The validation engine was already generic over (section, marker, id-pattern). This epic names that latent pattern instead of inventing new surface.
Model · 4 / 17

Addressable items & qualified ids

An addressable item is a structured line in a declared section carrying a self-id. Its qualified id is <scope>/<self-id> — globally unique, derived, never stored.
  • scope = an issue short-id, or @ for project scope.
  • Issue-scoped: c2a50aec/REQ-01  ·  project-scoped: @/INV-01.
  • The same self-id may exist under two scopes without collision.
Model · 5 / 17

An item kind is a six-tuple

(section, id-pattern, marker(s), link-namespace(s), scope, source-of-truth)
[item_kinds.requirement]
section         = "success_criteria"
id-pattern      = "[A-Z][A-Z0-9]*-[0-9]+"
markers         = ["[hard]"]
link-namespaces = ["satisfies"]
scope           = "issue"
source-of-truth = "markdown-first"
No kind name is hard-coded in engine logic — the engine reads the tuple, never a label. Kinds are config; the four shipped kinds are built-in defaults.

CLI · 7 / 17

Querying items: jit item

jit item list | show | resolve | search  [--kind <name>] [--json]
  • list — index every item across the repo (filter with --kind).
  • show / resolve — resolve one item by qualified id.
  • search — match on self-id or text.
Every subcommand supports --json — and failures emit a JSON error object too, honouring the machine-readable contract for agents.
CLI · 8 / 17

One list, all four kinds

$ jit item list
@/INV-01         [invariant]    All state changes are logged to events.jsonl.
@/INV-02         [invariant]    Dependency edges stay acyclic.
c2a50aec/D-01    [decision]     markdown is the source of truth.
c2a50aec/REQ-01  [requirement]  [hard] REQ-01: items are addressable.
c2a50aec/RISK-01 [risk]         index could drift from text.
Project-scope (@) and issue-scope (<short-id>) items surface through the same command — markdown-first and registry-first, one code path.

Invariants · 10 / 17

The invariant registry

Project-scoped invariants are registry-first: declared in .jit/invariants.toml, loaded at config time (even when no config.toml exists).

[[invariants]]
id          = "INV-01"
statement   = "All state changes are logged to events.jsonl."
kind        = "enforced"          # or "advisory"
enforced-by = "event-logging"     # optional: a rule or gate key
The registry is authoritative — there is no markdown index for invariants. The invariant kind is reserved as project-scoped + registry-first.
Invariants · 11 / 17

Projecting invariants into docs

jit invariant render [--json]  → writes the configured target
$ jit invariant render
Rendered 2 invariant(s) to .jit/invariants.md (separate-file mode)

## Project invariants
- **INV-01** [enforced] (enforced-by: `event-logging`): All state changes ...
- **INV-02** [advisory]: Dependency edges stay acyclic.
Two config-driven modes: a separate jit-owned file (default — never touches existing docs), or a delimited region of an existing file (byte-preserved outside the markers). No doc filename is hard-coded.
Links · 12 / 17

Links resolve by qualified id

A node carries a link label in a kind's namespace; it resolves to the addressed item across all four kinds:

satisfies:<scope>/REQ-01 · per:<scope>/D-01 · mitigates:<scope>/RISK-01 · enforces:@/INV-01
  • Existing label-coverage / criteria-to-check rules consume the same triple a kind expands to — shown compatible, not rewritten.
  • A label may be the legacy bare satisfies:REQ-01 or the qualified satisfies:<scope>/REQ-01; both credit coverage.
Links · 13 / 17

Dangling links surface in validate

An unresolvable qualified id is reported as a finding, not silently dropped — so broken links fail jit validate.

$ jit validate
✗ dangling-item-link: node 'a1b2c3d4' references 'satisfies:eed6750c/REQ-9'
  which resolves to no addressable item
The check derives link namespaces from the declared kinds — no kind name is hard-coded in the engine. It is a built-in pass, like coverage.
Invariants · 14 / 17

Bidirectional enforcement drift

jit invariant check [--json]  exits non-zero on any drift
$ jit invariant check
❌ [declared-but-unenforced] invariant 'INV-01' is enforced-by
   'event-logging', which is neither a loadable rule nor a known gate
❌ [enforced-but-undeclared] rule/gate 'default:orphan-leaf'
   is claimed by no invariant's enforced-by
Declaration-consistency, both directions. It runs in jit validate automatically when a registry is present, and stays dormant otherwise — so repos without invariants are unaffected.
Principles · 15 / 17

Three principles hold throughout

Single source of truth

Per-kind direction: markdown authoritative for issue kinds; registry authoritative for invariants. The index is a pure projection — never a hand-maintained duplicate.

Domain-agnostic, opt-in

Kinds are config-declared; no kind name lives in engine logic. The model fits any domain, not just software.

Graceful degradation

A line with no self-id is plain prose. You pay for addressing only where you want it.

Adopt · 16 / 17

Adopting it in your repo

  • Nothing required. The four kinds ship as built-in defaults — author ## Decisions / ## Risks sections and they index automatically.
  • Declare invariants in .jit/invariants.toml; run jit invariant render to mirror them into docs.
  • Tune kinds only if you need to — an explicit [item_kinds.X] declares the full six-tuple.
  • Query anytime with jit item list/show/search --json; wire links and let jit validate catch drift.
